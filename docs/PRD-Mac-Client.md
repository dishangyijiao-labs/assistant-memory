# 产品需求文档（PRD）：Assistant Memory Mac 客户端

**文档版本**：v1.0（已冻结，待执行）  
**决策状态**：已确认（2026-02-07）  
**目标读者**：产品、设计、研发、最终确认人  
**参考对标**：Claude Code `/insights` — 基于开发聊天记录生成报告与反馈  

---

## 一、愿景与目标

### 1.1 产品愿景

**一句话**：让开发者在一个本地 Mac 应用里，浏览、检索所有 AI 助手的聊天记录，并基于这些记录获得可执行的洞察与报告，提升开发效率与决策质量。

**价值主张**：
- **统一视图**：Cursor、Copilot、Cursor CLI、Claude Code、Codex、Gemini 等来源的会话与消息集中展示。
- **本地优先**：默认数据仅在本机处理；可选外部 API 接入需显式开启，满足隐私与合规。
- **从记录到洞察**：不止“能查”，还能“被总结、被分析、被反馈”——类似 Claude Code `/insights`，针对开发聊天记录生成报告与建议。

### 1.2 业务目标

| 目标 | 说明 | 可衡量方式 |
|------|------|------------|
| 提升检索效率 | 用户能快速找到某次对话或某段结论 | 从打开应用到找到目标会话的时长、搜索使用率 |
| 形成可复用知识 | 聊天记录可被浏览、复制、引用 | 消息查看数、复制/分享行为 |
| 从历史中产生洞察 | 基于多会话/多轮对话生成报告与反馈 | 报告生成次数、用户对报告有用性的反馈 |
| 保持零信任体验 | 不依赖账号，默认不上传数据 | 无账号登录、无遥测、外部 API 默认关闭且可审计 |

### 1.3 成功标准（MVP 后 3 个月）

- 日活/周活用户中，至少 **40%** 使用过「会话列表 + 消息详情」浏览。
- 至少 **25%** 用户使用过「洞察/报告」类功能至少一次。
- 至少 **30%** 用户查看过「评分模块」（效率分/稳定性分/决策清晰度分）。
- 用户反馈中「能找到历史对话」「报告有帮助」相关正面提及占比可统计并作为迭代依据。
- 达到 **公开 beta** 发布标准：无 P0 阻塞缺陷，核心链路（浏览/搜索/报告）可稳定完成。

## 二、用户与场景

### 2.1 目标用户

- **主要**：日常使用 Cursor / Copilot / Claude Code / Codex / Gemini 等工具的开发者（含独立开发者与小团队）。
- **次要**：技术负责人、导师——需要回顾团队或个人与 AI 的协作模式与产出，用于复盘或改进流程。

### 2.2 用户目标（Jobs to Be Done）

| 任务 | 描述 | 当前痛点 |
|------|------|----------|
| **回顾某次对话** | “上周和 Cursor 讨论过的那段登录逻辑在哪？” | 会话分散在各 IDE/CLI，难以按时间/项目聚合查找 |
| **复用结论** | “之前让 AI 总结的 API 设计我想再拿出来用” | 结论埋在长对话里，难以快速定位与复制 |
| **了解自己的使用模式** | “我最近主要在写哪类需求？卡在哪类问题上？” | 无结构化视图，只能凭印象 |
| **获得可执行反馈** | “基于我最近的对话，我该优先改进哪块代码/流程？” | 缺少类似 `/insights` 的汇总与建议 |

### 2.3 参考：Claude Code /insights 效果（目标对齐）

- 基于**开发聊天记录**做分析，而非泛化建议。
- 输出**报告/总结**：主题、任务类型、技术栈、问题模式等。
- 提供**可操作反馈**：例如“你常在某类错误上花时间，建议……”，“某模块被反复讨论，可考虑文档化”等。
- 界面简洁，一次操作即可生成并查看结果。

本产品的「洞察/报告」应达到：**针对用户自己的、本地的聊天记录，生成专属报告与反馈**，体验上可对标上述效果。

## 三、产品范围与阶段

### 3.1 范围概览

```
┌─────────────────────────────────────────────────────────────────────────┐
│  Phase 1（MVP）：会话 + 消息浏览                                          │
│  左侧 Sessions 列表 → 点击进入 → 右侧 Messages 列表，可查看与 AI 的完整对话  │
├─────────────────────────────────────────────────────────────────────────┤
│  Phase 2：检索与筛选增强                                                  │
│  全局搜索、按来源/时间/工作区筛选、消息内高亮                               │
├─────────────────────────────────────────────────────────────────────────┤
│  Phase 3：洞察与报告（类 /insights）                                       │
│  基于所选会话/时间范围/来源生成报告：总结、模式、可执行反馈                   │
└─────────────────────────────────────────────────────────────────────────┘
```

### 3.2 已确认产品决策（冻结）

| 项目 | 已确认结论 |
|------|------------|
| 报告生成方式 | **手动触发**（不做自动定时生成） |
| 报告默认范围 | **当前 workspace** |
| 模型策略 | **允许接入外部模型 API**（用户显式配置并同意后启用） |
| 报告结构 | **必须包含评分模块** |
| 行动项交互 | 首版**不做可勾选任务清单**，仅文本建议 |
| 发布目标 | 首版对外发布为 **Public Beta** |

以上决策为当前版本执行基线，可直接进入设计与研发排期。

## 四、功能需求详述

### 4.1 Phase 1：Mac 客户端 MVP — 会话与消息浏览

#### 4.1.1 应用形态

- **平台**：macOS 原生或跨平台桌面（如 Electron / Tauri），优先保证 Mac 体验。
- **数据**：复用现有 `~/.assistant-memory.db`（或 `ASSISTANT_MEMORY_DB_PATH`），与 CLI/Web 共用同一套 ingest 与 schema（sessions + messages + FTS）。
- **索引**：支持在客户端内触发「重新索引」（调用现有 index 逻辑），或提示用户使用 `assistant-memory index`；可选：后台定时/增量索引。

#### 4.1.2 布局与导航

- **整体布局**：
  - **左侧边栏**：Sessions 列表（主视图）。
  - **右侧主区域**：选中某 Session 后，展示该 Session 的 Messages 列表（时间线形式）。
  - 可选：顶部或侧边提供「设置 / 数据源 / 索引状态」入口。

- **左侧 — Sessions 列表**：
  - **每条 Session 展示**：来源（Cursor IDE / Copilot / Cursor CLI / Claude Code / Codex / Gemini）、工作区/项目标识（workspace 或 external_id 的简短展示）、最后活动时间、消息条数。
  - **排序**：默认按 `last_at` 降序（最近在前）。
  - **筛选**：按来源（source）筛选，如「全部 / Cursor IDE / Claude Code / …」。
  - **性能**：列表支持分页或虚拟滚动，避免一次性加载数千条。

#### 4.1.3 右侧 — Messages 详情

- **展示形式**：按时间顺序展示该 Session 内所有 messages；每条需明确区分 role：user / assistant / system（可用头像、背景色或标签区分）。
- **内容**：支持纯文本与基础 Markdown 渲染（代码块、列表、加粗等），便于阅读与复制。
- **交互**：复制单条消息、复制整场对话；可选：折叠长消息、展开/收起代码块。
- **无选中时**：右侧展示占位文案（如「在左侧选择一场会话以查看对话」）。

#### 4.1.4 数据与同步

- **读取**：仅读现有 DB；不修改现有 ingest 逻辑，仅通过既有 schema 读取 sessions 与 messages。
- **索引**：提供「立即索引」或「检查索引状态」；若 DB 为空或过旧，引导用户执行索引或说明数据来源（哪些路径被扫描）。
- **多源**：继续支持当前所有 source（cursor, cursor-cli, copilot, claude-code, codex, gemini），在 Session 列表中统一展示并可用 source 筛选。

#### 4.1.5 Phase 1 验收标准

- [ ] 在 Mac 上可安装并打开客户端。
- [ ] 左侧展示 Sessions 列表，支持按来源筛选，默认按最近活动排序。
- [ ] 点击某 Session 后，右侧展示该 Session 下全部 Messages，角色与时间清晰可辨。
- [ ] 支持从现有 `assistant-memory` DB 读取数据；可在应用内或通过 CLI 触发索引后看到新会话。
- [ ] 无崩溃、无数据错乱；大量会话（如 500+）下列表仍可流畅滚动。

---

### 4.2 Phase 2：检索与筛选增强

- **全局搜索**：在 Sessions 或 Messages 内容上做关键词搜索（复用 FTS），结果可为「匹配的 Sessions」或「匹配的 Messages」并定位到对应 Session + 消息。
- **筛选增强**：按时间范围（如最近 7 天、本月）、按 workspace 或 external_id 模糊筛选。
- **高亮**：在消息正文中高亮搜索词。
- **优先级**：先做「全局搜索 + 结果跳转到 Session/Message」，再做时间/workspace 筛选与高亮。

---

### 4.3 Phase 3：洞察与报告（类 Claude Code /insights）

#### 4.3.1 目标效果

- 用户在当前 workspace 上点击「生成报告」后**手动触发**分析；可选切换到其他范围（时间区间、来源、多 Session）。
- 系统基于这些会话中的 **user + assistant 内容** 做分析（本地规则/统计 + 可选外部模型 API），产出：
  - **总结**：主题分布、常见技术栈、任务类型（修 bug、写功能、重构、解释等）。
  - **模式**：反复出现的文件/模块、常见错误或卡点。
  - **可执行反馈**：如「你多次询问 X，建议写一份内部文档」「某类问题耗时较多，可考虑加测试」等。
  - **评分模块（必选）**：效率分、稳定性分、决策清晰度分（0-100）。

#### 4.3.2 功能要点

- **触发方式**：仅手动触发（按钮/快捷键），不做自动周报任务。
- **范围选择**：按时间、按 Session 多选、按来源；默认 **当前 workspace**。
- **报告类型**（可分批实现）：
  - **摘要报告**：会话数、消息数、主要来源、时间分布。
  - **主题/任务报告**：从消息内容中抽取或推断主题（关键词、简单分类）。
  - **反馈报告**：基于规则或模型生成 3–5 条可执行建议（风格上对齐 /insights），首版为文本列表，不做勾选清单。
  - **评分报告**：输出效率/稳定性/决策清晰度三个维度分数，并给出扣分依据。
- **展示**：独立「报告」视图或面板，可复制、可导出（Markdown/文本）。
- **模型接入**：默认本地分析；允许用户在设置中接入外部模型 API（如 OpenAI/Anthropic/其他兼容接口），需显式启用并展示数据发送范围。

#### 4.3.3 与 Claude Code /insights 的对应

| /insights 能力 | 本产品对应 |
|----------------|------------|
| 基于开发聊天记录分析 | 仅使用用户本机 DB 中的 sessions/messages |
| 生成总结与模式 | 摘要报告 + 主题/任务报告 |
| 给出可操作建议 | 反馈报告（规则或模型，支持外部 API 可选接入） |
| 一次操作出结果 | 「选择范围 → 生成报告 → 查看/复制」 |

## 五、用户故事（汇总）

### Phase 1

1. 作为开发者，我希望在 Mac 上打开一个应用就能看到所有 AI 会话列表，以便快速找到某次对话。
2. 作为开发者，我希望按来源（如 Cursor IDE、Claude Code）筛选会话，以便只关注某一工具的历史。
3. 作为开发者，我希望点击某条会话后在右侧看到完整对话（含我和 AI 的每条消息），以便回顾或复制内容。
4. 作为开发者，我希望应用使用我已经用 CLI 索引好的数据，无需重复配置，以便开箱即用。

### Phase 2

5. 作为开发者，我希望用关键词搜索会话或消息内容，并能跳转到对应消息，以便精确定位某段讨论。
6. 作为开发者，我希望按时间范围筛选会话，以便只看最近一周或本月的记录。

### Phase 3

7. 作为开发者，我希望针对选定的会话或时间范围生成一份「洞察报告」，以便了解自己的使用模式与改进点。
8. 作为开发者，我希望报告里包含可执行建议（类似 Claude Code /insights），以便直接改进工作流或代码质量。
9. 作为开发者，我希望报告给出可解释的评分结果，以便量化我当前的协作质量与改进优先级。

## 六、信息架构（IA）与关键界面

### 6.1 信息架构

```
Assistant Memory Mac 客户端
├── 首页 / 会话列表（左侧）
│   ├── 筛选：全部 | Cursor IDE | Copilot | Cursor CLI | Claude Code | Codex | Gemini
│   ├── 排序：最近活动（默认）
│   └── 列表项：来源 · 工作区/项目 · 时间 · 消息数
├── 会话详情（右侧）
│   └── 消息时间线：user / assistant / system，时间，内容（支持 Markdown）
├── [Phase 2] 搜索
│   └── 全局搜索 → 结果列表 → 点击跳转 Session + Message
└── [Phase 3] 洞察 / 报告
    ├── 选择范围（时间 / 来源 / 会话）
    ├── 生成报告
    └── 报告视图（摘要 / 主题 / 反馈）
```

### 6.2 关键界面草图（文字描述）

- **主界面**：左侧约 280–320px 宽 Sessions 列表；右侧剩余宽度为 Messages 时间线；顶部可放「索引状态」「设置」。
- **Session 列表项**：图标（来源）+ 标题（workspace 或截断的 external_id）+ 副标题（时间 + 消息数）。
- **Message 气泡/卡片**：左侧 user，右侧 assistant；system 可用中性样式；时间戳可选显示在悬停或每 N 条合并显示。

## 七、非功能需求

### 7.1 性能

- Sessions 列表首次加载 < 2s（500 条量级）；选中 Session 后 Messages 首屏 < 1s。
- 列表滚动 60fps，无明显卡顿。

### 7.2 隐私与安全

- 所有数据来自本机 DB，不默认上报任何内容。
- 外部模型 API 为**可选能力**，默认关闭；开启前需明确告知发送内容范围并获得用户同意。
- 若未来有「上传匿名统计」等，必须可选且明确告知。

### 7.3 兼容性

- 最低 macOS 版本需明确（建议 macOS 12+ 或与 Electron/Tauri 选型一致）。
- 与现有 `assistant-memory` CLI/Web 共用同一 DB 与 schema，不破坏现有使用方式。

### 7.4 可访问性

- 支持键盘导航（列表、焦点切换）；关键按钮与列表项支持 VoiceOver 等读屏。

## 八、技术约束与建议（供研发参考）

- **数据层**：继续使用现有 SQLite + FTS；客户端以只读或「读 + 写索引状态/配置」为主，不改变现有 sessions/messages 表结构。
- **桌面技术选型**：Electron 或 Tauri 均可；若强调体积与性能可倾向 Tauri；若强调与 Web 复用可倾向 Electron。
- **Phase 3 报告**：首版即包含摘要、反馈、评分三部分；推理链路支持规则+统计，并预留外部模型 API Provider 抽象（本地/云端可切换）。
